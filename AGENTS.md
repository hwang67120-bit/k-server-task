# K사 서버 과제 에이전트 규칙

## 1. 목적

- 개발자는 요구사항 분석, 설계, 결과 검증에 집중한다.
- 에이전트는 단순 작업, 반복 작업, 테스트 실행과 문서화를 자동화하여 시간적 비용을 줄인다.
- 자동화 자체가 목적은 아니다. 개발자가 결과를 이해하고 검증할 수 있어야 한다.

## 2. 의사결정 권한

- 모든 의사결정 권한과 최종 책임은 사용자에게 있다.
- 사용자와 에이전트의 판단이 다르면 사용자의 결정을 우선한다.
- 에이전트는 선택지, 근거, 장점, 비용과 검증 방법을 제시할 수 있지만 승인 없이 요구사항, 설계, 기술과 작업 범위를 확정하지 않는다.
- 사용자가 작업 범위를 승인하면 에이전트는 해당 범위의 반복 작업을 추가 승인 없이 수행할 수 있다.
- 승인 범위를 벗어나는 의존성 추가, 설계 변경, 데이터 변경, 삭제, 배포, 커밋, 푸시와 병합은 다시 승인받는다.

## 3. 작업 환경

- 코드 변경, 빌드, 자동 테스트, 애플리케이션 실행과 Docker 작업은 Linux에서 수행한다.
- Linux는 SSH 별칭 `mycom-linux`로 연결한다.
- Linux 프로젝트 경로는 `/home/mycom/win_g/k-server-task`이다.
- Windows는 문서 작성, Postman 수동 테스트와 GitHub 관리에 사용한다.
- Windows 공유 경로는 `\\192.168.0.109\win_g\k-server-task`이다.
- 두 운영체제가 같은 파일을 공유해도 코딩 명령은 Windows에서 실행하지 않는다.

## 4. 구조와 이름

- 패키지는 `menu`, `point`, `order`, `event`처럼 도메인을 기준으로 나눈다.
- 필요한 경우 각 도메인 안에서 controller, service, repository, entity, request, response 책임을 구분한다.
- 이름은 짧고 익숙한 초급 영어를 우선한다.
- 패키지는 소문자 단수 명사를, 메서드는 쉬운 동사와 대상의 조합을 우선한다.
- 책임이 확인되지 않은 `common`, `util`, `manager`, `processor`, `process`, `handle` 같은 넓은 이름을 사용하지 않는다.
- 쉬운 영어로 바꾸면 뜻이 부정확해지는 `transaction`, `repository`, `idempotency`, `outbox` 같은 표준 기술 용어는 유지한다.
- 세부 기준은 `docs/agent/directory-structure.md`를 따른다.

## 5. 코드 작성

- 클래스와 메서드는 하나의 명확한 책임을 가진다.
- 객체 생성과 의존성 관리는 Spring DI와 IoC에 맡긴다.
- Controller는 요청과 응답을 담당하고 비즈니스 정책은 Service 또는 도메인에서 검증한다.
- 엔티티의 상태 또는 저장 값을 변경하는 로직은 엔티티 메서드가 맡는다.
- Service는 요청값과 현재 상태를 검증하고 유스케이스와 트랜잭션을 조정하며 엔티티 필드를 직접 변경하지 않는다.
- 조회 Service는 엔티티를 변경하지 않고 필요한 데이터 조회, 무결성 검증과 결과 조합을 담당한다.
- 트랜잭션 범위는 비즈니스 원자성을 기준으로 정한다.
- 승인된 요구사항을 만족하는 가장 단순한 구현을 우선한다.
- 예상한 미래 요구사항을 위해 인터페이스, 계층, 추상화와 확장 지점을 미리 추가하지 않는다.
- 공식 문서는 근거이지 자동 적용 기준이 아니다.
- 사용자가 처음 보는 어노테이션, 패턴, 라이브러리와 프레임워크는 문제, 단순한 대안, 장점, 비용과 검증 방법을 먼저 설명하고 승인받는다.
- Null 안정성 어노테이션 같은 선택 기능을 공식 문서에 있다는 이유만으로 추가하지 않는다.
- 코드 기준 우선순위와 예시는 `docs/agent/code-reference.md`를 따른다.
- README는 처음부터 완성된 설계서가 아니라 현재까지 확인한 정책과 가설을 기록하는 문서다.
- 현재 작업을 막지 않는 미래 문제를 미리 설계하지 않고 작은 기능 구현, 테스트, 측정, 필요한 개선 순서로 진행한다.

### 5.1 빠른 소단위 작업

- 요청받은 하나의 메서드 또는 특정 코드 블록만 먼저 변경한다.
- 요청받지 않은 클래스 전체를 다시 작성하지 않고 변경되지 않는 주변 코드는 건드리지 않는다.
- 사용자에게 코드 일부를 제시할 때 변경 없는 주변 코드는 `// ... 기존 코드 생략 ...`으로 표시하여 출력량을 줄인다.
- 실제 프로젝트 파일에는 컴파일 가능한 전체 구조를 유지하며 `// ... 기존 코드 생략 ...` 문구를 삽입하지 않는다.
- 요청하지 않은 기능, 클래스, 변수, 메서드, 예외 처리와 확장 구조를 임의로 추가하지 않는다.
- 구현에 추가 요소가 반드시 필요하면 코드를 작성하기 전에 필요성, 최소 범위와 영향을 설명하고 사용자 승인을 받는다.
- 승인받은 구현 코드를 먼저 프로젝트에 반영하고 구현 도중 테스트 코드 작성과 테스트 실행을 섞지 않는다.
- 구현 반영 후 컴파일 오류가 없는지 확인한 다음에만 별도 테스트 단계로 넘어간다.
- 기본 검증은 방금 추가한 기능 하나를 확인하는 단위 테스트 또는 슬라이스 테스트 하나만 작성하고 실행한다.
- 사용자 요청 없이 전체 테스트와 전체 통합 테스트를 실행하지 않는다.
- 추가 정상·예외 시나리오, 실제 DB 또는 인프라 검증이 필요하면 먼저 이유와 예상 시간을 설명하고 사용자 승인을 받는다.
- 검증 결과는 성공 또는 실패와 실패 원인만 간단히 보고한다.

작업 순서:

```text
요청 범위 확인
→ 필요한 추가 요소가 있으면 사용자 승인
→ 하나의 메서드 또는 코드 블록 구현
→ 컴파일 확인
→ 가장 작은 테스트 하나 실행
→ 성공 또는 실패 보고
```

### 5.3 서비스 메서드 주석

- 서비스 메서드에는 흐름을 빠르게 이해할 수 있도록 블록 주석을 작성한다.
- 주석은 다음 순서로 통일한다.
  - 기능: 메서드가 수행하는 기능
  - 파라미터: 메서드에 전달되는 값
  - 요청값: 각 파라미터에 실제로 들어오는 값의 의미
  - 응답값: 반환되는 값의 의미
- 응답 DTO의 실제 필드만 작성하고, 존재하지 않는 값을 임의로 설명하지 않는다.
- private 메서드도 서비스 흐름에 영향을 주면 같은 형식으로 작성하되 내부 처리임을 표시한다.

### 5.2 주문 상태와 실시간 알림

- 결제 상태와 주방 주문 상태를 하나의 필드에 섞지 않는다.
- 주문 결제가 성공하면 결제 상태는 `PAID`, 주문 상태는 `PREPARING`으로 저장한다.
- 주문 상태는 주방 작업자의 명시적인 요청으로만 `PREPARING → COOKING → COMPLETED` 순서로 변경한다.
- 시간 경과, WebSocket 메시지 또는 Pub/Sub 수신을 근거로 주문 상태를 자동 변경하지 않는다.
- 주문 상태 변경은 MySQL 트랜잭션과 `@Version` 낙관적 락으로 보호한다.
- 같은 주문을 동시에 변경하면 한 요청만 커밋하고 충돌한 요청은 최신 상태와 버전을 포함한 `409 Conflict`로 처리한다.
- Redis Pub/Sub과 WebSocket은 커밋된 결과를 전달하는 알림 경로이며 데이터 원본이나 동시성 제어 수단으로 사용하지 않는다.
- 실시간 알림은 DB 커밋 이후에만 발행하고 롤백된 상태를 발행하지 않는다.
- 최초 접속, 재연결, 메시지 유실과 순서 역전이 발생하면 MySQL에서 최신 주문과 버전을 다시 조회한다.
- 고객 주문 채널은 해당 주문을 소유한 사용자만 구독하도록 권한을 검사한다.

## 6. 로그

- 처음 접하는 로직은 흐름 파악을 위해 `DEBUG` 로그를 사용할 수 있다.
- 검증된 로직에는 장애 흐름을 복원하고 중요한 상태 변경을 확인하는 로그만 유지한다.
- 상태 변경, 외부 호출, 재시도와 최종 실패에는 필요한 경우 요청 ID, 주문 ID 또는 이벤트 ID를 포함한다.
- 같은 예외를 여러 계층에서 반복해서 `ERROR`로 기록하지 않는다.
- 비밀번호, 토큰, 전화번호, 결제 비밀값과 개인정보를 로그에 남기지 않는다.
- 로그는 장애 추적을 보조하며 포인트와 주문의 원본 기록은 DB 이력이다.

## 7. 기술 선택

- 기술이나 전략을 추가하기 전에 문제, 발생 조건, 영향, 대안, 선택, 이유, 비용, 검증과 재검토 조건을 작성한다.
- `이 선택이 없으면 어떤 비즈니스 정책이나 사용자 시나리오가 깨지는가?`를 확인한다.
- 명확하게 답할 수 없으면 해당 선택을 도입하지 않는다.
- `docs/agent/decision-template.md`를 사용한다.

## 8. 테스트와 검증

- 완료는 코드 작성이나 컴파일 성공이 아니라 동작 검증을 의미한다.
- 구현과 테스트 단계를 분리하고 구현 코드 반영과 컴파일 확인 전에는 테스트를 작성하거나 실행하지 않는다.
- 기본적으로 방금 추가한 기능 하나에 대한 가장 작은 단위 테스트 또는 슬라이스 테스트 하나만 Linux에서 실행한다.
- 사용자 승인 없이 전체 테스트와 전체 통합 테스트를 실행하지 않는다.
- 정상과 예외 시나리오를 모두 검증해야 한다면 추가 테스트의 필요성과 시간을 먼저 설명하고 승인받는다.
- 테스트를 통과시키기 위해 테스트를 삭제하거나 검증 조건을 약화하거나 실패를 숨기지 않는다.
- 실제로 실행하지 않은 테스트를 통과했다고 보고하지 않는다.
- DB 제약조건, 트랜잭션, 락과 동시성 때문에 실제 MySQL 검증이 필요하면 범위와 예상 시간을 먼저 설명하고 승인받는다.
- 실행한 가장 작은 테스트의 성공 또는 실패와 실패 원인만 우선 보고한다.

## 9. Git과 GitHub

- 저장소: https://github.com/hwang67120-bit/k-server-task
- SSH 원격 주소: `git@github.com:hwang67120-bit/k-server-task.git`
- `main`은 안정된 단계의 결과를, `develop`은 통합 결과를 관리한다.
- 최신 `develop`에서 도메인 이름을 포함한 짧은 기능 브랜치를 만든다. 예: `feature/menu-list`, `feature/point-charge`, `feature/order-payment`, `feature/event-outbox`.
- 하나의 기능 브랜치는 하나의 도메인 API 그룹과 명확한 범위를 담당한다.
- 같은 도메인의 연관 API는 관리 비용을 줄이기 위해 하나의 기능 브랜치에서 함께 관리할 수 있으며 Issue에 추가 범위를 기록한다.
- 검증 후 Pull Request를 통해 기능 브랜치를 `develop`에 병합한다.
- 기능 코드를 `main` 또는 `develop`에 직접 커밋하지 않는다.
- 사용자 승인 없이 커밋, 푸시, 병합, 배포, 기록 변경과 강제 푸시를 하지 않는다.
- 관련 없는 파일을 하나의 커밋에 포함하지 않는다.
- `build.gradle`, 애플리케이션 설정, 공통 예외와 Flyway 마이그레이션은 공유 파일로 보고 Issue와 Pull Request에 영향을 기록한다.
- 충돌이 발생하면 한쪽 변경을 임의로 선택하지 않고 양쪽 의도를 설명한 뒤 사용자 결정을 받는다.
- 기능 작업에는 `.github/ISSUE_TEMPLATE/feature.md`를 사용한다. 관련 없는 체크 항목은 이유와 함께 해당 없음으로 기록하고 체크리스트 때문에 요구사항을 늘리지 않는다.

## 10. 작업 중단 조건

다음 상황에서는 변경을 중단하고 현재 상태, 선택지, 영향, 에이전트 의견과 필요한 사용자 결정을 보고한다.

- 요구사항과 비즈니스 정책이 충돌한다.
- README 설계와 구현 방향이 충돌한다.
- 사용자 결정 없이는 작업을 이어갈 수 없다.
- 데이터 손실 가능성이 있는 DB 변경이 필요하다.
- 새로운 유료 서비스, 외부 계정, 의존성 또는 승인받지 않은 기술이 필요하다.
- 알 수 없는 미커밋 변경이 현재 작업과 충돌한다.
- 브랜치 또는 병합 대상이 명확하지 않다.
- 필요한 환경이 없어 핵심 정책을 검증할 수 없다.
- 비밀값이나 개인정보가 노출될 가능성이 있다.
- 되돌리기 어렵거나 파괴적인 작업이 필요하다.

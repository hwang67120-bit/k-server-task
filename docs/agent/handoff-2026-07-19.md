# K-server-task 인수인계 문서

작성일: 2026-07-19
현재 브랜치: `feature/order-payment`

## 1. 프로젝트와 작업 원칙

- Java 17, Spring Boot 3.5.16을 사용한다.
- 개발·빌드·테스트는 Linux에서 수행한다.
- Windows는 문서·Postman·GitHub 관리에 사용한다.
- 기능 브랜치는 `develop`에서 생성하고 검증 후 PR로 병합한다.
- 사용자가 요청하지 않은 기능과 미래 확장 구조는 추가하지 않는다.
- 구현과 테스트를 분리하고, 구현 후 컴파일 확인 다음 최소 테스트만 실행한다.
- 서비스 메서드는 기능·파라미터·요청값·응답값 순서의 블록 주석을 사용한다.

## 2. 완료된 기능

### 메뉴

- 메뉴 목록 조회 API 구현
- 카테고리는 enum이 아니라 변경 가능한 데이터로 취급
- 메뉴 상세 조회 API 구현
- 옵션의 사이즈는 `REGULAR`, `LARGE` enum
- 온도는 `HOT`, `ICE` enum
- 커피콩 종류는 문자열
- 시럽 여부는 boolean
- 가격은 원화 정수 금액이므로 `long`
- 메뉴 서비스 메서드에 흐름 주석 추가

### 포인트

- 포인트 충전 도메인 구현
- 요청값: 사용자 ID, 전화번호, 결제 ID, 충전 금액
- 응답값: 사용자 ID
- 사용자 ID와 전화번호 일치 검증
- Mock 결제 완료 여부·결제 ID·금액 검증
- `paymentId` unique 제약과 비관적 락 조회 사용
- 완료 상태면 기존 응답을 반환하는 멱등성 처리
- `PointAccount.addPoint(long)` 추가
- 포인트 충전·사용 내역 조회 서비스 구현
- 공통 예외 메시지에 결제 연결 실패 추가: `인터넷이 끊겼습니다.`
- 결제 검증은 최대 3회 재시도
- 현재 포인트 충전 단위 테스트 성공

### 주문

- 주문·결제 API 구현
- `orderToken`, `requestHash` 기반 멱등성 값 반영
- 주문 생성 시 결제 상태와 주방 주문 상태를 분리
- 결제 성공 시 결제 상태 `PAID`, 주문 상태 `PREPARING`
- 포인트 차감과 주문 처리를 트랜잭션으로 묶음
- 주문 브랜치: `feature/order-payment`

### 이벤트·인기 메뉴

- `OrderCompletedEvent` DTO 추가
- `DailyProductSales` 엔티티 추가
- 상품별 일일 판매 수량과 주문 횟수 누적
- `OrderEventProducer` 추가
- `OrderEventConsumer` 추가
- Kafka 이벤트 기반 일일 판매 집계 구현
- 인기 메뉴 조회를 위한 `ProductSalesProjection`과 repository 쿼리 추가
- Kafka 토픽 설정 클래스는 아직 보류

### 정책·문서

- 공통 시간 관리용 BaseTimeEntity 사용
- 글로벌 예외 처리 및 ErrorCode 사용
- 주문 상태 정책: `PREPARING -> COOKING -> COMPLETED`
- 주문 상태는 주방의 명시적 요청으로만 변경
- `@Version` 낙관적 락 사용
- Redis Pub/Sub·WebSocket은 알림 경로이며 DB 원본이 아님
- 연결 복구 시 DB에서 최신 상태·버전을 재조회
- 외부 프로젝트의 WebSocket 재연결 예시는 검토했지만 현재 프로젝트에는 실제 적용하지 않음

## 3. 현재 테스트 상태

현재 `PointChargeServiceTest`는 Mockito 기반 단위 테스트다.

- 정상 충전 성공
- 사용자 정보 불일치
- 결제 연결 실패 후 3회 재시도
- 첫 결제 검증 실패 후 재연결 성공

```bash
bash ./gradlew test --tests '*PointCharge*' --no-daemon
```

## 4. 아직 하지 않은 작업

### 최우선

- 실제 포인트 충전 동시성 테스트
- 주문 결제 동시성·멱등성 테스트
- 트랜잭션 일관성·롤백 테스트
- Postman 직접 검증
- TIL 작성

### 동시성 테스트의 주의점

현재 포인트 테스트는 Mockito 단위 테스트라 실제 DB의 `@Version` 증가와 충돌을 검증하지 못한다.

실제 검증에는 다음이 필요하다.

- Testcontainers MySQL 실행
- 포인트 계좌 테이블 스키마 또는 Hibernate 테스트 스키마 확인
- 실제 repository를 이용한 두 트랜잭션 실행
- 한 요청의 성공과 다른 요청의 optimistic lock 충돌 검증

현재 `PointChargeService`에는 낙관적 락 충돌 재시도·예외 변환 로직이 없으므로, 테스트 작성 전에 이 정책을 먼저 확정해야 한다.
